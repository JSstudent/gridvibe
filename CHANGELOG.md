# Changelog

All notable changes to GridVibe will be documented in this file.

## Unreleased

- **Static assets always revalidate (edits show up without a cache clear):** `terminals.js`/CSS and the other static files are cache-busted only by `?v={{ version }}`, which is the fixed app version — so an edited file kept the same URL and a long-lived client cache (notably the desktop WebView2 profile, whose cache survives app restarts) served the **stale** copy, making UI changes appear not to take effect (e.g. an updated split-button size gate still greyed out where the old build did). Flask now sets `SEND_FILE_MAX_AGE_DEFAULT = 0`, so the client revalidates static assets on every load and always picks up the current file (negligible cost over localhost for this single-user tool).
- **Explicit split direction + folding pane-header actions:** the terminal pane header's **Close split** button (`⊟`) was removed — a split is now closed the same way any pane is, with the existing **×** close button, which reflows the freed space into the neighbour sharing the longest border. Splitting is now **two explicit controls** — **Split into side-by-side panes** and **Split into stacked panes** — so the direction is your choice, not an inferred guess; each greys out independently (with an axis-specific tooltip) when that axis no longer fits, the workspace is at its configured pane cap (up to 16), or the window is too narrow. Splitting is no longer limited by the base launcher layout or a pane's split history — any pane can be split repeatedly until the next split would make a terminal too small (the previous rule blocked re-splitting already-split panes in 3+-pane layouts, so a freshly split pane often couldn't be split again), and each split produces two fully **independent** panes with no shared parent/child linkage, so resizing or closing one never has surprising side effects on the other. The base layout grid was also given more vertical resolution so **stacked (horizontal) splits are no longer capped at a single level**, and the per-pane minimum height for a stacked split was lowered (a split pane must keep ~4 rows rather than 8) so a normal wide pane can actually be stacked two or three deep — you can now build, say, a tall column of three stacked terminals beside one large pane, gated only by the (smaller) character-size minimum. A stacked split needs each half to keep at least 4 rows after its own header, so any pane taller than roughly 200px can be split into a column; panes shorter than that are genuinely too small and stay greyed. To stop the header buttons from clipping on tiny panes, every action except **Close** now folds into a **⋯ overflow menu** the moment the header runs out of room (measured from the header's real width, so a pane below roughly 1/8 of the surface collapses first); **Close** always stays directly reachable, and the menu dismisses on an outside click, on **Escape**, or after you pick an action.
- **Richer Explorer Source highlighting and precise Git diffs (`docs/source_diff_analysis.md`, Phases 1–2):** the Explorer **Source** viewer now syntax-highlights the whole document in one pass with a locally vendored, pinned Highlight.js 11.9.0 build (common languages plus `dos`, `powershell`, and `dockerfile`), so multiline comments, strings, template literals, and embedded HTML `<script>`/`<style>` keep their coloring across line boundaries instead of resetting each line, and languages like Go, Python, HTML, CSS, JavaScript, and TypeScript distinguish far more token categories (functions, types, attributes, tags, decorators) than the previous five-token lexer. The **Diff** viewer now renders through a locally vendored, pinned diff2html 3.4.48 build with character-level intraline emphasis and LCS line matching, so a one-character change highlights only the characters that changed and related old/new lines align correctly instead of FIFO pairing; truncated diffs (the backend still caps at 256 KiB / 4,000 lines) now show a non-blocking banner so the shown patch is never mistaken for the whole change. Token colors are explorer-scoped and theme-aware, line numbers/search/font zoom/scroll restoration/Markdown fold all still work, and the diff remains a fixed 50/50 split with original source lines intact plus an independent, viewport-sticky horizontal scrollbar for each side. Every path falls back gracefully — oversized or unsupported source files render as plain text, and if either library is unavailable or fails the viewer reverts to the previous handwritten renderer. All assets are pinned, licensed, documented in `web/static/vendor/README.md`, and served locally (no CDN).
- **Git sidebar handles new files individually:** untracked directories are expanded into their actual files in **Changes**, so their rows open and stage correctly instead of appearing as a non-clickable `? folder/` entry. Each untracked file now also has a discard control with an explicit permanent-deletion confirmation; the API accepts only the selected file, while **Discard All** remains tracked-only and continues to preserve every untracked file.
- **Internal — `terminals.js` split (N6, `docs/terminals_js_split_plan_2026-07-23.md`):** the 13,510-line `terminals.js` was decomposed into cohesive classic-script files via three behaviour-preserving, move-only extractions, each loaded after `shared.js` and before `terminals.js`: `terminal-icons.js` (SVG button icons), `voice-input.js` (mic capture, recording overlay, hold/push-to-talk), and `explorer-viewer.js` (file-type classifier, syntax highlight, Git diff, source/Markdown render, image/mermaid viewer, tabbed viewer, breadcrumb, per-tab view/scroll state, saved-tab persistence). All shared state, socket init, boot, and top-level key listeners remain in `terminals.js`, which shrank to ~7,190 lines. No user-visible behaviour changed; every moved line is byte-identical to its former self. `ExtractedFrontendAssetsTestCase` now locks in all four files and their load order.
- **Guardrail audit fixes (`docs/guardrail_audit_2026-07-22.md`):** the launcher's **Save workspace & restart** button works in the native desktop window again — it confirmed through `window.confirm`, which the native WebView2 window blocks, so the button silently did nothing exactly where the restart bridge exists; the launcher page now ships the same in-page `genericConfirmModal` shell as the terminals workspace, and both the restart and the browser-mode **Close** confirmations (plus the close-failure alert) go through it and the inline status line. The remaining emoji glyphs on terminal pane-header buttons (📁 explorer toggle, 🌐 browser preview, 🎤 voice, ☾/☀ explorer theme, and the `>_` text glyph) were replaced with stroke-style `currentColor` SVG icons matching the rest of the toolbar. Internally, `session_status` broadcasts now snapshot the session under the manager lock and emit after releasing it, so one slow client can no longer stall session management. All four fixes are locked in by a new `GuardrailAuditFixesTestCase`, and the regression guardrails in `CLAUDE.md`/`AGENTS.md` were tightened accordingly (no `window.*` dialogs anywhere, no emits under any shared lock, no further growth of `terminals.js`).
- **Save workspace & restart from the launcher:** the launcher action bar gained a restart icon button between **Check for updates** and **App settings** that captures the workspace and relaunches GridVibe in one action. It confirms first (live shells do not survive a restart), then saves through the same `POST /api/runtime-state/save` path as the terminal workspace's **Save Workspace** — an empty workspace (`409`) is treated as "nothing to save" rather than an error, so the restart still proceeds — and relaunches via the existing native `restart_application()` bridge (the same one `checkForUpdates` uses after applying an update). In browser mode (or with no native bridge) there is nothing to relaunch, so it saves the workspace and asks you to restart GridVibe manually; save success/failure and restart status are surfaced on the launcher's inline status line.
- **Workspace restore replays the current saved preset ("latest preset wins"):** a restored session group that was launched from a *named* saved session now relaunches from that preset's **current** config rather than the shape frozen into the snapshot when the group first launched. Previously the restore banner replayed the captured launch fields verbatim, so editing a saved session (or running **Save All Sessions**) after a group was already open had no effect on restore until you re-imported the session — the workspace was a photo of what launched, and the photo never re-read the preset even though it stored the link. Now **Save All Sessions** followed by **Save Workspace** restores the edited state directly: on restore, each preset-backed group re-fetches `GET /api/saved-sessions/<id>` and rebuilds its panes from that config (layout, directories, commands, agent + Auto mode, and the saved SSH password). Ad-hoc groups with no saved session, a deleted/missing preset, or the blank built-in default still replay the snapshot verbatim (the default's config carries no real host/directory, so it is never treated as a preset). The launcher's launch-button expansion and the restore path now share one `buildSessionsFromConfig()` helper so both build panes identically.
- **Workspace restore hardening (autosave + explicit save, multi-workspace-ready schema):** restore-after-restart was rewritten from an event-driven snapshot (rewritten on nearly every group change — the source of the flaky collapsed restores and the stray *"Session HH:MM:SS"* pane) to a deterministic model with exactly **two writers**: a backend **autosave daemon thread** and an explicit save. The autosave interval is a new global setting, **Auto-save workspace every N min** (1–15, default 5) in App Settings, re-read each tick so slider changes apply without a restart. The terminal workspace top bar gained a **Workspace…** dropdown (sharing the renamed neutral `.app-menu*` component with **Sessions…**) whose **Save Workspace** item captures immediately via `POST /api/runtime-state/save` and confirms with the saved label and time; it is disabled while the workspace is empty. Group create/split/close events and window teardown no longer write the snapshot at all — losing up to one interval of unsaved shape is the deliberate trade for never persisting a transient shape. The restore offer is now **permanent** (the 12 h age gate is gone), the banner names the saved workspace and its age ("Restore *buildserver-01* — saved 12 min ago"), and **Dismiss** only hides the banner — the saved slot is preserved and simply overwritten by the next capture, so an idle launcher or a fresh restart can never wipe it. `runtime_state.json` moved to schema v2: one slot per workspace id with a stable, human-facing label (real group name → host → directory basename → "Workspace" — never a bare timestamp, and restore no longer mints `Session HH:MM:SS`), stored origin (`auto`/`manual`), and the same per-session launch fields as before (passwords still never persisted); a one-time v1→v2 migration keeps an existing user's restore offer. The schema, endpoints (`workspace_id`-keyed GET/DELETE), and persistence are already per-workspace, so the planned multi-workspace upgrade (named workspaces, Save As / Import / Rename / Delete) is purely additive.
- **Save All Sessions:** the terminal workspace's **Sessions…** dropdown gained a **Save All Sessions** item that saves every open session tab in one action, not just the active one. Each tab is saved in place through the same workspace-save path as **Save Session** — a tab already backed by a saved session updates it, an unsaved tab is created — so the saved-session library and the restart snapshot stay in sync across every tab. Background tabs are captured from their cached workspace view (live DOM for the visible tab), the per-tab saved-session cross-window broadcast still fires for each, and the status line reports how many sessions were saved (and how many failed, if any). Empty tabs are skipped rather than counted as failures.
- **Explorer tree & Markdown fold polish:** the Files-tree **open-in-a-new-tab** control no longer wears the same `+` glyph as the Git **Stage** button — it now carries a distinct ↗ ("open") glyph so the two controls can't be mistaken for one another (its tooltip, theme-aware surface, and behaviour are unchanged). In the Markdown **Source** viewer, holding **Alt** while clicking a heading's fold arrow now collapses (or expands) **every heading at that same level** in one action instead of just the clicked section — the new state mirrors the clicked heading, so Alt+collapsing an expanded `##` folds all `##` sections at once, and Alt+clicking a folded one unfolds them all. The fold arrow's tooltip now advertises the Alt shortcut.
- **Explorer image viewer:** clicking an image file (`.png`, `.jpg`/`.jpeg`, `.gif`, `.webp`, `.bmp`, `.ico`, `.svg`, `.avif`, `.apng`) now shows it in a read-only inline viewer instead of failing with "appears to be binary". The `/file` metadata route reports `preview_type: "image"` (no content) and the bytes stream from a new `GET /api/explorer/<id>/image` route that serves them inline with the right MIME type, 25 MB-capped and root-confined like download/preview; SVGs are served under a locked-down `Content-Security-Policy` (`default-src 'none'; sandbox`) plus `nosniff` so embedded script can't run even on direct navigation (`<img>` never executes it anyway). The viewer reuses the existing tab strip, breadcrumb, and download button — so images open in Preview and pinned tabs, survive refresh, and persist like any other tab — over a theme-aware checkerboard that keeps transparent images legible in both light and dark; the file size and pixel dimensions show in the header.
- **Open the current Explorer location in the system file manager:** the Explorer bar gained a button next to the Files-tree and Git toggles (in front of the path breadcrumb) that opens the host OS file manager on whatever the pane is currently showing — a listed directory opens directly, an open file is revealed (selected) inside its folder — so you can copy, rename, or delete outside GridVibe's read-only panes. It works for both the Preview and pinned open-file tabs and is deliberately isolated from the explorer's browsing logic: a new `POST /api/explorer/<id>/reveal` route resolves the path through the same root-confinement used by download/preview and only launches the local file manager (`explorer` / `open -R` / `xdg-open`), never mutating files. The button is shown for local panes only (SSH explorer panes omit it, since a remote path has no meaning for the server's file manager). Its icon is now the standard "open external" glyph with hover/active/focus feedback so it reads clearly in both themes instead of sitting flat.
- **Markdown viewer improvements:** heading folds in Markdown Source view are now stored per tab, survive tab swaps and saved-session restores, and are discarded when the underlying document identity changes. Fenced `mermaid` blocks render as diagrams through a locally vendored Mermaid 11.15.0 bundle with strict security and inline parse errors. **Preview** can be toggled with `Ctrl+Shift+V` (also shown in its tooltip), and the Markdown appearance menu now includes the distinct curated font stacks offered by Terminal settings without equivalent duplicate choices.
- **Explorer Preview tab keeps its own path across tab swaps and saves:** the Preview tab's browsed directory previously lived only in pane-global state, so swapping to a pinned tab and back left the Preview showing nothing (its path label had been overwritten by the other tab). The Preview tab record now carries its own separated path — the directory it browses (`dirPath`) alongside the file it shows — so returning to the tab re-browses exactly where you left it, scroll position included (same OD-4 identity check as file tabs; a changed listing opens at the top instead of a stale offset). The separated path also persists: workspace **Save Session** (dropdown) and launcher session saves now store the Preview tab's shown file and browsed directory under the reserved `__preview__` tab-view key (allowlist-validated on the backend like the other tab views, round-tripped untouched by the launcher), so relaunching a saved session reopens the Preview content — an active pinned tab still wins the viewer on restore, with the Preview content one click away. The initial-load path also no longer fires a bare root-directory fetch for connected explorer panes (it now goes through the viewer entry point like every other first show), so a slow root response can no longer resolve after the restore and clobber the Preview tab's path back to the top directory. The terminal-close rebuild snapshot (5.a) carries `dirPath` through as well.
- **Explorer directory browsing from the tree + breadcrumb (follow-up 2.d):** clicking a directory in the Files tree now also opens that directory in the Preview tab as a navigable listing, so you can drill in from the main pane (the expand/collapse toggle is unchanged; clicking the directory the Preview tab already shows just toggles its expansion, and navigating no longer force-expands the target row — expansion stays a tree-click decision). The explorer bar's path label became a lightweight **breadcrumb**: every ancestor segment (root included) is clickable and browses that directory in the Preview tab — the up-one-level / jump-to-ancestor navigation that replaces the Back button removed in follow-up 2.a. The breadcrumb renders for directory listings, open files, and commit diffs alike.
- **Explorer tab-strip interactions (follow-up 2.g):** pinned tabs can now be **dragged to reorder** (the permanent Preview tab keeps the first slot — it cannot be dragged and nothing can be dropped ahead of it; the saved-session tab order follows the reordered sequence), **middle-clicking** a pinned tab closes it (same guard as the × — the Preview tab is never closed), and **double-clicking the Preview tab** promotes the file it shows to a pinned tab that keeps the exact view mode, scroll position, and zoom (the already-rendered viewer is handed over without a re-fetch; if the file is already pinned, that tab is activated instead). Clicking a tab that is already shown is now a no-op instead of a redundant re-fetch.
- **Restore-after-restart no longer collapses to a single stray pane, and restored agents keep Auto mode:** two bugs made the "Previous workspace found — Restore?" banner unreliable. (1) `runtime_state.json` was rebuilt from the *live* session set on every group change, so any moment the live set was transiently small — e.g. after closing the session window while the launcher stayed open, then closing a leftover scratch pane — collapsed the snapshot to that one unnamed `Session HH:MM:SS` pane, which is all the next restart then offered. The snapshot is now captured authoritatively **at teardown** (session-window close, app exit, or Ctrl+C in browser mode) from the full live workspace, and group-close events **prune only the closed group** from the persisted file instead of rebuilding from a possibly-collapsed live set (closing the last group still clears it). (2) Restoring replays each pane through the normal launch path, which ran agent **preflight** and cleared any command whose binary probe merely *errored* (`check_failed`) — common with a cold detection cache right after a restart — so the agent (and its Auto-mode flag) silently vanished. Restore now replays commands verbatim (skips preflight-clearing) and lets a genuinely missing binary surface in the pane's own error placeholder. Complements the earlier multi-token-flag fix below (a separate Auto-mode-on-restore cause).
- **Agent Auto mode survives workspace import + codex flag fixed:** two auto-mode gaps left from follow-up 7.b. First, importing a saved session from the terminal workspace's **Sessions… → Import Session** menu silently dropped the toggle — `buildSavedSessionLaunchPayload()` rebuilt each session field-by-field and never carried `agent_auto_mode` (the launcher's direct-launch and the restart-restore paths already did), so imported agents started as the bare command. The workspace import payload now carries the toggle, matching every other launch path. Second, codex's `--full-auto` flag was removed from the interactive CLI (deprecated in Codex CLI v0.128) and now errors with `unexpected argument '--full-auto'`; the registry auto-mode flag for codex is now the sandboxed **Auto preset** — **`--sandbox workspace-write --ask-for-approval on-request`** — so edits and commands run automatically inside the workspace and only genuinely unsafe actions prompt. Supporting this required loosening `_agent_auto_mode_flag()` to accept multi-token option/value flags (previously any whitespace was rejected); the validator still refuses shell metacharacters, so a registry typo can never smuggle a second command onto the launch line. This also fixes Auto mode being ignored on restart-restore, which was the same rejected-flag symptom rather than a separate data-loss bug.
- **Explorer tab state persists in sessions (follow-up 2.f):** saving a session (or the auto restart snapshot) now also captures, per Explorer tab, the active **view mode** (Source/Preview/Diff) and **scroll position** (stored as a fraction of scroll height, with the content-identity hash so a changed file reopens at its default view instead of a stale offset), alongside the already-persisted open tabs and active tab. The **Markdown appearance** (reading-surface preset + font, ISSUE-2026-033) rides the same three persistence paths — saved presets, workspace Save Session, and `runtime_state.json` — so relaunching a saved session on another machine or after clearing browser storage restores the appearance it was saved with (applied once per session, so an appearance changed mid-session is never clobbered by a pane rebuild; when a session carries no appearance the local preference applies as before). The per-tab **editor zoom** persists too — each pinned tab's font size comes back after a relaunch, and the permanent Preview tab's zoom is saved under a reserved key (values are clamped to the 10–24 zoom bounds; tabs at the default size store nothing). All new fields are allowlist-validated on the backend and the launcher round-trips them without editing. Closes ISSUE-2026-033.
- **Closing a terminal no longer resets Explorer tab state (follow-up 5.a):** closing a terminal pane forces a grid rebuild, which previously dropped the displayed Explorer tab's Source/Preview/Diff mode and scroll position (only the open tabs and previewed file survived). The close snapshot now carries every tab's full view record — mode, scroll metrics, content identity, per-tab zoom, and the Preview tab's sticky mode preference — and reattaches them before the rebuilt pane re-fetches its content, so the same identity check from 2.e restores exactly what was on screen.
- **Explorer tabs keep their view mode and scroll position (follow-up 2.e):** swapping between Explorer file tabs no longer resets each tab's state — a Markdown tab stays in **Preview**, a diff tab stays in **Diff**, and every tab returns to the scroll position it was left at. Each tab record now carries its own view snapshot (mode plus scroll stored as a fraction of scroll height, clamped on restore), captured on the way out of a tab and restored on the way back — including the diff panel, whose content loads asynchronously. If the file changed in the meantime (a tail-updated log, a re-fetch with new bytes), the stale snapshot is ignored and the tab opens at its default view instead of a misleading offset. The Preview tab's directory browsing position is preserved the same way. This per-tab model is the shared foundation for persisting tab state into saved sessions and for surviving terminal-close rebuilds.
- **Explorer Preview tab isolation + per-tab zoom (follow-up 2.e refinement):** the permanent **Preview** tab is now fully isolated from pinned tabs — clicking a file in the Files tree or Git sidebar always loads it into the Preview tab, even when that file is already open in a pinned tab, so pinned tabs keep their own mode, scroll, and zoom untouched (previously, clicking a pinned-open file under Changes/Staged Changes hijacked that tab into diff mode). The editor font-size zoom is now stored per tab instead of shared across all of them, and the Preview tab also remembers your **Source/Preview** preference across different files — opening another Markdown file stays in Preview, while files without a preview fall back to Source (scroll resets on a new file, and Diff remains an explicit per-view choice). Tab-state captures are additionally guarded by a rendered-tab marker, so viewing a file's diff in the Preview tab can no longer overwrite the saved mode/scroll of a pinned tab showing the same file.
- **Git sidebar bulk actions + post-action refresh (follow-up Wave 3):** the **Changes** header gained two compact bulk controls — **Stage All** (`POST /api/explorer/<id>/git/stage-all`, `git add --all` at the repo root) and **Discard All** (`POST /api/explorer/<id>/git/discard-all`), which after an in-page confirmation restores only tracked, non-conflicted worktree changes via `git restore --worktree`, preserving staged content and never deleting untracked files (no `git clean`). Both are disabled while a Git action is busy or when there is nothing to act on. Every worktree-mutating Git action — per-row stage/unstage/revert, commit, and the new bulk pair — now also reloads the Files tree and re-fetches the open file/diff in place (scroll preserved), so the tree and stale diff hunks no longer linger after a commit. The branch **Publish** confirmation also moved from `window.confirm` (blocked in the native WebView2 window) to the in-page confirm shell. Closes ISSUE-2026-032 and ISSUE-2026-034.
- **Broadcast highlight follows real focus (follow-up 6.a):** with Broadcast typing on, clicking into dead space or an Explorer/browser pane now drops the accent ring from every terminal pane instead of leaving it lit; focusing a terminal again re-lights all participating panes if broadcast is still on, or just the clicked pane if broadcast was switched off in the meantime.
- **Terminal font presets + per-session scope (follow-up 7.c):** the App Settings Font Family free-text field became a dropdown of curated monospace presets — each option previews in its own typeface — with a **Custom…** option that reveals the original free-text input. Saved font and size now apply to the **currently active session** by default — every terminal pane of the visible workspace tab, recorded as an in-memory per-session override that survives tab switches, pane rebuilds, and splits; a new **Apply to all active sessions** checkbox pushes both to every session in one save, including the cached hidden session tabs. The scope rides the app-config broadcast as a one-shot `apply_scope` modifier and is never persisted to `config.json`; new sessions keep launching with the saved global font.
- **Agent Auto mode actually launches with the flag (follow-up 7.b):** the per-terminal **Auto mode** toggle was silently dropped on the direct-launch path — `launchSessions()` rebuilt each session payload field-by-field and never included `agent_auto_mode`, so agents always started as the bare command (`kimi`, `claude`, …) regardless of the toggle. The payload now carries the toggle, and `kimi --auto-approve`, `claude --enable-auto-mode`, `codex --full-auto`, `copilot --allow-all-tools`, and `kilo --yolo` launch as shown. The toggle's help text now shows the registry description for every flag-carrying agent (`Launches as "kimi --auto-approve". Launches with all actions auto-approved (alias of --yolo).`) instead of only the bare composed command, and **Kilo CLI** gained its Auto mode entry (`--yolo`). OpenCode intentionally has no Auto mode toggle: upstream opencode has no skip-permissions CLI flag — its permissions are config-file driven (`opencode.json`), so there is no flag to register.
- **Explorer & agents polish (follow-up Wave 2):** the file preview cap was raised from 1 MiB to 10 MiB — files in between (like that 1.2 MB JSON-Lines export) now open in full instead of truncating — while previews above ~2 MiB render as plain text (no syntax highlighting) so the viewer stays responsive; the log tail policy and the precise "Showing the last/first N of M" messaging are unchanged. `go.mod`, `go.sum`, `go.work`, and `go.work.sum` are now recognized for editor preview (previously rejected with "format is not supported"), with the frontend classifier mirrored for matching highlighting. Binary detection no longer rejects a valid UTF-8 file whose 4 KiB sample happens to end mid-character — a trailing partial sequence is deferred only when more content follows, while NUL bytes, invalid sequences inside the sample, excess control bytes, and genuinely truncated short files are still rejected (local and SFTP explorers alike). And **Kimi Code CLI** joined the agent registry (`kimi`, Auto mode `--auto-approve`) with the official install-script/uv install paths for Windows, Linux/WSL, and detect-only SSH targets. Closes ISSUE-2026-035.
- **Explorer & Markdown polish (follow-up Wave 1):** the Files tree row **+** (open in new tab) button now follows the light/dark theme via the shared explorer tokens instead of inheriting a mismatched search-button surface; the vestigial single-file **Back** button was removed from the tabbed viewer's file header (navigation lives in the permanent Preview tab); the Markdown appearance popover gained a fourth reading-surface preset, **Slate** (key `vscode`) — a VS Code-style dark gray surface with near-white text that persists like the other presets; and Markdown heading lines in the **Source** viewer are now coloured (heading-only tokeniser reusing the fence-aware heading map, accent-token colour, no new dependency).
- **Terminal settings in App Settings:** the App Settings modal gained a **Terminal** section exposing the config that previously required hand-editing `config.json` — terminal font family, font size (6–48), and max sessions (1–16). Values are validated and clamped on save and on config load, persist through `/api/app-config` into `config.json`, and font changes apply to every open terminal window immediately over the existing cross-window app-config contract (max sessions applies to new launches). The modal body also scrolls properly again: the Save/Cancel row is pinned below a scrollable settings area at every window width, so enabling voice input (or the new fields) no longer pushes content under the action buttons. Closes ISSUE-2026-029 and ISSUE-2026-031.
- **Per-agent Auto mode toggle:** agent-mode terminals gained an **Auto mode** checkbox that appears only when the selected built-in agent has a registered auto-mode flag in `agent_registry.json` (`claude --enable-auto-mode`, `codex --full-auto`, `copilot --allow-all-tools`), with help text showing the exact command it will launch. The choice is per terminal and round-trips through launcher drafts, saved sessions, workspace saves, and the restart-restore snapshot, while the stored startup command stays the bare agent key — the flag is appended once at launch time from validated registry metadata, so agent preflight detection keeps working and custom commands are never modified. Closes ISSUE-2026-013.
- **Surgical terminal close:** closing a terminal pane in a split layout is now precise. When geometry allows, exactly one neighbour — the pane sharing the longest border with the closed pane — expands into the freed space instead of every touching pane growing together, and panes outside the expansion keep their manually resized proportions (previously a close reset all panes to uniform widths/heights). Sibling panes in the same group also keep their state across the close: Explorer panes retain their open file/preview, expanded Files tree, and Git sidebar; browser panes keep their URL — where before closing one terminal reset every other pane in the group to a fresh listing/reload. (Browser scroll still resets, since a cross-origin iframe cannot be re-scrolled.) Closes ISSUE-2026-022 and ISSUE-2026-027.
- **Large log previews keep the newest lines:** opening an oversized log in an Explorer pane now retains the **end** of the file instead of the beginning, so current failures and recent activity are visible without an external tail tool. `.log` files keep their last ~1 MiB (trimmed to a clean line and UTF-8 boundary so the preview never starts mid-character or mid-line); every other oversized text file keeps its opening ~1 MiB as before. The file header now reports exactly what you're seeing — `Showing the last <N> of <M>` for logs, `Showing the first <N> of <M>` otherwise — in place of the generic "Preview truncated". Works for both local and SSH/SFTP explorers. Closes ISSUE-2026-020.
- **Markdown preview presentation:** the Explorer Markdown **Preview** got a polished, theme-aware document surface — clearly differentiated `h1`–`h6` (with dividers under the top levels), visible nested list markers, styled links and rules, tables with a header tint and zebra rows, and task-item/image handling. GitHub-style admonition callouts are now supported: a blockquote that starts with `[!NOTE]`, `[!TIP]`, `[!IMPORTANT]`, `[!WARNING]`, or `[!CAUTION]` renders as a coloured callout block with a stroke-style icon and label (emitted server-side *after* sanitization, so the callout classes/icons never widen the HTML sanitizer and callout bodies stay cleaned).
